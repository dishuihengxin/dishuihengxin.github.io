Pod

Pod 是 kubernetes 集群中操作的最小单元，但不是容器状态下的最小组件，一个 Pod 可以拥有一组 container 资源，做为集群中运行的一个进程集合。同一个 Pod 中各 container 共享存储、网络等，但 container 之间又是一个相互独立。

  • 网络

每个 Pod 创建时都被分配一个唯一的 IP 地址，Pod 中所有容器共享同一网络空间，包括 IP 地址和端口。内部容器之间相互无阻碍的通信，但与外部进行通信时，需要打通 Pod 网络与宿主机网络的通信，通常只需要映射端口就可以解决这个问题。

  • 存储

可以指定 Pod 共享多个 Volume资源，对于共享的 Volume 资源，所有 container 都可以直接访问，Volume 可以作为 Pod 的持久化存储资源，防止 Pod 重建后文件丢失的问题。

如何使用 Pod

你很少会直接在kubernetes中创建单个Pod。因为Pod的生命周期是短暂的，用后即焚的实体。当Pod被创建后（不论是由你直接创建还是被其他Controller），都会被Kuberentes调度到集群的Node上。直到Pod的进程终止、被删掉、因为缺少资源而被驱逐、或者Node故障之前这个Pod都会一直保持在那个Node上。

注意：重启Pod中的容器跟重启Pod不是一回事。Pod只提供容器的运行环境并保持容器的运行状态，重启容器不会造成Pod重启。

Pod不会自愈。如果Pod运行的Node故障，或者是调度器本身故障，这个Pod就会被删除。同样的，如果Pod所在Node缺少资源或者Pod处于维护状态，Pod也会被驱逐。Kubernetes使用更高级的称为Controller的抽象层，来管理Pod实例。虽然可以直接使用Pod，但是在Kubernetes中通常是使用Controller来管理Pod的。

Pod 生命周期：

  • Pending(挂起)： Pod 已被 kubernetes 集群接受，正在创建过程中，有一个或多个 container 尚未创建，等待时间包括调度 pod 的时间和下载镜像的时间。
  • Running(运行中)： 该 Pod 已经绑定到某个或多个节点上，所有的 container 已经创建完成并在运行中。
  • Successed(成功)： Pod 中 container 已经被成功终止，且不会自动重启。
  • Failed(失败)： Pod 中容器已经被终止，且至少有一个容器因为终止失败，以非 0 状态退出。
  • Unknown(未知)： 因为某些原因无法获取到 Pod 的状态，通常都是因为网络通信失败的问题。
